# 5.1 Celery 동작 흐름

## 개요

서버는 `start.sh` 하나로 3개의 프로세스를 순차적으로 기동한다.

```
[Beat] ---(매일 10:00 태스크 발행)--→ [Redis: batch:celery 큐]
                                           │
                                 [Worker] ←─┘ (큐에서 메시지 꺼내서 실행)
                                           │
                                           ├── DB 크롤링 결과 저장
                                           └── [Redis: batch:celery-task-meta-xxx] 결과 저장
```

```
start.sh 실행
│
├── Step 1: Worker 기동
│   ├── celery_app.py 로드 → Celery 객체 생성 (메모리)
│   ├── rank_tasks.py 로드 → 태스크 등록 (메모리)
│   ├── Redis 연결 → batch:_kombu.binding.* 키 생성 (큐 선언)
│   └── batch:celery 큐 polling 대기 (큐 자체는 아직 존재하지 않음)
│
├── Step 2: Beat 기동
│   ├── celery_app.py 로드 → Celery 객체 생성 (메모리)
│   ├── RedBeat 스케줄러 초기화
│   ├── Redis 연결 → batch:redbeat:* 키 생성 (스케줄 등록, 분산 락)
│   └── crontab 시간에 태스크 메시지를 batch:celery 큐에 발행
│
└── Step 3: FastAPI 기동
    └── Celery와 무관하게 API 서버 운영
```

```bash
# api/start.sh
celery -A app.tasks.celery_app worker --concurrency=1 -l info &   # Step 1
celery -A app.tasks.celery_app beat -l info &                      # Step 2
exec uvicorn app.main:app --host 0.0.0.0 --port 8000              # Step 3
```

## 핵심 개념

### Producer / Consumer

Celery에서 **Producer**는 태스크 메시지를 큐에 넣는(발행하는) 주체, **Consumer**는 큐에서 꺼내 실행하는 주체다.

| 역할 | 이 프로젝트에서 누가? | 설명 |
|------|---------------------|------|
| **Producer** | **Beat (RedBeat)** | 유일한 producer. `beat_schedule`의 crontab 시간에 태스크 메시지를 큐에 넣음 |
| **Consumer** | **Worker** | `batch:celery` 큐에서 메시지를 꺼내 태스크 실행 |
| FastAPI | **producer 아님** | 코드 어디에서도 `.delay()`, `.apply_async()` 등을 호출하지 않음 |

> 일반적인 Celery 프로젝트에서는 웹 서버(FastAPI/Django 등)가 API 요청을 받아 `.delay()`로 태스크를 발행하는 producer 역할을 겸하는 경우가 많다.
> 이 프로젝트에서는 Beat의 스케줄에 의해서만 태스크가 생성되므로, Beat이 유일한 producer다.

### Redis 키 접두사 규칙

| 설정 | 값 | 적용 대상 |
|------|-----|----------|
| `broker_transport_options.global_keyprefix` | `batch:` | Kombu(브로커) 관련 키 전체 |
| `result_backend_transport_options.global_keyprefix` | `batch:` | 태스크 결과 저장 키 |
| `redbeat_key_prefix` | `batch:redbeat` | RedBeat 내부 키 (`schedule`, `lock`, `statics` 등) |

> RedBeat 엔트리 키는 `{redbeat_key_prefix}{엔트리명}` 형태로 콜론 구분자 없이 붙는다.
> 예: `batch:redbeatcrawl-all-active-trackings`

## 기동 순서

### Step 1: Celery Worker

```
celery -A app.tasks.celery_app worker --concurrency=1 -l info &
```

1. **`app.tasks.celery_app` 모듈 로드**
   - `celery_app.py`가 import되면서 `Celery("announce_go", ...)` 객체가 Python 메모리에 생성
   - `include=["app.tasks.rank_tasks"]` 설정에 의해 `rank_tasks.py`도 import → `@celery_app.task`로 등록된 태스크가 워커에 등록됨
   - 이 시점에는 아직 Redis 연결 없음

2. **Redis Broker 연결 및 큐 선언**
   - 워커가 실제로 시작되면서 `settings.redis_url`로 Redis에 연결
   - Kombu가 큐 바인딩 정보를 Redis에 등록

3. **대기 상태 진입**
   - `batch:celery` 큐를 polling하며 태스크 메시지를 기다림

| 관련 파일 | 역할 |
|----------|------|
| `api/app/tasks/celery_app.py` | Celery 앱 인스턴스 생성 및 설정 |
| `api/app/tasks/rank_tasks.py` | `crawl_all_active_trackings` 태스크 정의 |
| `api/app/tasks/__init__.py` | `celery_app` re-export (`from app.tasks.celery_app import celery_app`) |

### Step 2: Celery Beat

```
celery -A app.tasks.celery_app beat -l info &
```

1. **`app.tasks.celery_app` 모듈 로드** (Step 1과 동일한 모듈, 별도 프로세스)

2. **RedBeat 스케줄러 초기화**
   - `beat_scheduler="redbeat.RedBeatScheduler"` 설정에 의해 RedBeat이 스케줄러로 동작
   - RedBeat은 스케줄 정보를 Redis에 저장하므로, 여러 인스턴스가 떠도 스케줄이 중복 실행되지 않음

3. **스케줄 등록**
   - `celery_app.conf.beat_schedule`에 정의된 스케줄을 Redis에 기록:
     ```python
     "crawl-all-active-trackings": {
         "task": "app.tasks.rank_tasks.crawl_all_active_trackings",
         "schedule": crontab(hour=10, minute=0),  # 설정값에 따라 다름
     }
     ```

4. **주기적 태스크 발행 대기**
   - 다음 실행 시각(score)까지 대기
   - 해당 시각이 되면 `crawl_all_active_trackings` 태스크 메시지를 `batch:celery` 큐에 넣음
   - Step 1의 워커가 이 메시지를 꺼내서 실행

| 관련 파일 | 위치 |
|----------|------|
| `api/app/tasks/celery_app.py` | `celery_app.conf.beat_schedule` — 스케줄 정의 |
| `api/app/tasks/celery_app.py` | `beat_scheduler="redbeat.RedBeatScheduler"` — 스케줄러 지정 |
| `api/app/core/config.py` | `CRAWL_SCHEDULE_HOUR=10`, `CRAWL_SCHEDULE_MINUTE=0` — 기본 스케줄 시간 |

### Step 3: FastAPI

```
exec uvicorn app.main:app --host 0.0.0.0 --port 8000
```

- Celery와는 **완전히 별개** 프로세스
- `app/main.py`의 FastAPI 앱이 기동됨
- API 요청을 처리하는 역할이며, Celery 태스크를 직접 실행하지 않음
- Celery와의 접점 없음 (현재 코드 기준, API에서 태스크를 수동 트리거하는 엔드포인트는 없음)

## Redis 키 레퍼런스

### 기동 시 생성되는 키

`start.sh` 실행 직후 (태스크 발행 전) 생성되는 전체 키:

| Step | 키 | 타입 | 생성 주체 |
|------|----|------|----------|
| 1 | `batch:_kombu.binding.celery` | Set | Worker |
| 1 | `batch:_kombu.binding.celery.pidbox` | Set | Worker |
| 1 | `batch:_kombu.binding.celeryev` | Set | Worker |
| 2 | `batch:redbeat:lock` | String | Beat |
| 2 | `batch:redbeat:schedule` | Sorted Set | Beat |
| 2 | `batch:redbeat:statics` | Set | Beat |
| 2 | `batch:redbeatcrawl-all-active-trackings` | Hash | Beat |

> `batch:celery`(List) 큐 자체는 이 시점에 **생성되지 않는다.**
> Beat이 태스크를 발행해야 메시지가 들어가면서 List가 생기고, Worker가 꺼내가면 비어서 Redis가 자동 삭제한다.

### 태스크 실행 시 생성되는 키

Beat이 스케줄 시각에 태스크를 발행하고, Worker가 실행을 완료하면:

| 키 | 타입 | 설명 |
|----|------|------|
| `batch:celery` | List | 태스크 메시지가 들어오면 생성, Worker가 꺼내면 사라짐 (일시적) |
| `batch:celery-task-meta-<task-id>` | String (JSON) | 태스크 실행 결과. `task-id`는 UUID (예: `batch:celery-task-meta-a1b2c3d4-...`) |

### RedBeat 키 상세

| 키 | 타입 | 설명 |
|----|------|------|
| `batch:redbeat:schedule` | Sorted Set | 등록된 스케줄 목록. score = 다음 실행 시각 (Unix timestamp) |
| `batch:redbeatcrawl-all-active-trackings` | Hash | 개별 스케줄 엔트리 상세 (태스크명, crontab, 마지막 실행 시각 등) |
| `batch:redbeat:lock` | String | Beat 인스턴스 중복 실행 방지용 분산 락 (TTL: `redbeat_lock_timeout=300`초) |
| `batch:redbeat:statics` | Set | RedBeat이 관리하는 정적 스케줄 엔트리 목록 |

> **`batch:redbeat:schedule`의 실제 데이터 예시:**
> ```json
> { "value": "batch:redbeatcrawl-all-active-trackings", "score": 1772585999 }
> ```
> score `1772585999` = 2026-03-04 10:00 KST (다음 실행 예정 시각)

### _kombu.binding.* 키 상세

**키 네이밍 패턴:**

```
{global_keyprefix}_kombu.binding.{exchange_name}
```

즉 `batch:` (global_keyprefix) + `_kombu.binding.` (Kombu 내부 접두사) + `{exchange 이름}` 구조다.

| Redis 키 | Exchange 이름 | 용도 |
|----------|--------------|------|
| `batch:_kombu.binding.celery` | `celery` (기본 exchange) | 일반 태스크 메시지 라우팅 |
| `batch:_kombu.binding.celery.pidbox` | `celery.pidbox` | Worker 원격 제어 (inspect, ping 등) |
| `batch:_kombu.binding.celeryev` | `celeryev` | 이벤트 모니터링 (task-started, task-succeeded 등) |

**Set 멤버 형식:**

각 키는 Redis Set이며, 멤버는 아래 형식의 문자열이다:

```
{routing_key}\x06\x16\x06{queue_name}
```

`\x06\x16\x06`은 Kombu가 내부적으로 사용하는 구분자(delimiter)다.

예를 들어 `batch:_kombu.binding.celery`의 멤버:

```
celery\x06\x16\x06celery
```

이것은 **"routing key가 `celery`인 메시지는 `celery` 큐로 보내라"** 는 바인딩 규칙이다.

**왜 필요한가:**

Redis는 RabbitMQ와 달리 exchange/binding 개념이 네이티브로 없다.
Kombu가 **Redis Set으로 exchange→queue 바인딩을 시뮬레이션**하는 것이다.

```
Beat이 태스크 발행
  → Kombu가 exchange 이름으로 batch:_kombu.binding.celery Set 조회
  → 멤버에서 routing_key 매칭되는 queue_name 확인
  → 해당 큐(batch:celery List)에 메시지 LPUSH
```

## 부록: 인력 시장 비유로 이해하는 Celery

Celery의 구조는 **인력 시장(인력 사무소)**의 동작 원리와 거의 완벽하게 일치한다.

| Celery 구성 요소 | 인력 시장 비유 | 설명 |
|-----------------|--------------|------|
| **Worker** | 인부들 | 인력 사무소에 새벽같이 나와서 대기하고 있는 분들. 먼저 와 있어야 "사람 보내달라"는 요청이 올 때 즉시 출발할 수 있다. |
| **Redis** | 사무소 게시판 | 오늘 해야 할 일들이 적히는 대기 명단. 인부가 없는데 일거리만 적히면 게시판이 꽉 차고 난리가 난다. |
| **Beat** | 작업 반장 | "자, 10시 됐으니 저기 A 현장 크롤링 작업 시작하세요!"라고 일거리를 던져주는 사람. |
| **Kombu Binding** | 사무소의 행정 절차 | 인부들이 오면 "나는 도배 전문이다", "나는 타일 전문이다"라고 등록을 한다. 그래야 반장이 "도배 일거리"가 생겼을 때 누구한테 시켜야 할지 알 수 있다. |

### 실행 순서가 중요한 이유

- **인부(Worker)가 먼저 올 때:** 반장(Beat)이 "일 시작!" 하는 순간 대기하던 인부가 바로 투입된다. (매끄러움)
- **반장(Beat)이 먼저 소리칠 때:** 인부도 없는데 반장 혼자 게시판에 "일하세요!"라고 쪽지만 계속 붙여놓는 상황. 나중에 인부들이 한꺼번에 도착하면, 쌓여있는 쪽지를 보고 한꺼번에 몰려서 허둥지둥하게 된다.

> 결국 **"일할 준비가 된 사람을 먼저 확보한다"**는 철학이 `start.sh`의 실행 순서(Worker → Beat → FastAPI)에 반영된 것이다.
